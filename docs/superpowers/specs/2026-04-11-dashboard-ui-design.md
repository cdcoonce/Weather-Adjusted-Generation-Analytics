# WAGA Dashboard UI — Design Spec

**Date:** 2026-04-11
**Status:** Approved, pending implementation plan
**Author:** Claude + Charles Coonce

## Purpose

Build an interactive dashboard that (a) lets Charles explore the WAGA pipeline's
generation and weather analytics data and (b) serves as a portfolio showcase
demonstrating the full data engineering stack — from Snowflake to in-browser
visualization — on charleslikesdata.com.

The dashboard has two audiences with overlapping needs:

1. **Charles** — wants an operational view of asset performance, weather
   correlations, and data quality across the 10-asset mock fleet.
2. **Portfolio visitors** — should see a polished, interactive data product
   that visually matches the portfolio site and tells a coherent story about
   the pipeline's outputs.

## Goals

- Exploratory interactivity (asset selection, date filtering, drill-down,
  cross-chart filtering)
- Visual consistency with charleslikesdata.com (Poppins font, neutral palette,
  editorial tone)
- Zero marginal infrastructure cost beyond what the WAGA project already pays
  (GitHub, Dagster Cloud, Snowflake)
- Fully automated deploys for both data refresh and app code changes
- Clean separation between pipeline (owns data) and UI (reads pre-computed
  exports)

## Non-Goals

- Live Snowflake queries from the browser
- Real-time streaming updates (daily refresh is fine)
- Authentication or private views — everything on the dashboard is public
- Self-service BI features (ad-hoc query builder, custom metrics, CSV export)
- Mobile-first layout — desktop-primary is acceptable for a portfolio piece

## Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│  WAGA Dagster Pipeline (existing)                           │
│  RAW -> dbt staging -> marts -> Polars analytics            │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  NEW: waga_dashboard_export asset                           │
│  - Downstream of mart_asset_performance_daily and           │
│    mart_asset_weather_performance                           │
│  - Projects to UI-required columns only                     │
│  - Writes JSON files locally                                │
│  - Pushes JSON to portfolio repo via GitHub API             │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  Portfolio repo (charleslikesdata.com)                      │
│  /dashboard/                                                 │
│    ├── index.html        (Panel + Pyodide bootstrap)        │
│    ├── app.js            (generated by panel convert)       │
│    ├── worker.js         (Pyodide web worker)               │
│    ├── portfolio.css     (theme)                            │
│    └── data/                                                 │
│        ├── manifest.json                                     │
│        ├── assets.json                                       │
│        ├── daily_performance.json                            │
│        └── weather_performance.json                          │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  Visitor's browser                                          │
│  - Downloads static HTML + JS                               │
│  - Pyodide bootstraps (~10MB, browser-cached)               │
│  - Panel app runs entirely in-browser                       │
│  - Fetches data/*.json, renders interactive charts          │
└─────────────────────────────────────────────────────────────┘
```

**Key architectural decisions:**

- **Panel + Pyodide (`panel convert --to pyodide-worker`)** runs the dashboard
  entirely in the browser. No backend server, no cold starts, no infrastructure
  to maintain, $0 marginal cost.
- **Pre-aggregated JSON exports** shaped for the UI — not raw mart data. The
  export asset projects mart columns down to only what charts actually use.
- **Cross-repo push via GitHub Contents API** (not git clone/push) — works from
  Dagster Cloud serverless, single HTTPS request, idempotent.
- **Two decoupled deploy flows:** data refreshes on every pipeline run, app
  code refreshes on every push to `main` that touches `src/weather_analytics/dashboard/`.

## Components

### 1. Dagster Export Asset

**Location:** `src/weather_analytics/assets/analytics/dashboard_export.py`

**Asset name:** `waga_dashboard_export`

**Dependencies:** `mart_asset_performance_daily`, `mart_asset_weather_performance`

**Responsibilities:**

- Query both marts via `WAGASnowflakeResource.get_connection()`
- Project columns down to the UI subset (~15 columns per mart)
- Build four JSON payloads (see Data Contract below)
- Write all four files to `dashboard_exports/` locally (for debugging and
  local builds)
- Push all four files to the portfolio repo's `dashboard/data/` directory via
  the GitHub Contents API, in a single commit
- Raise `dagster.Failure` on any API error with the full response logged
- Emit asset metadata: row counts, byte sizes, commit SHA

**New config entries:**

- `WAGA_PORTFOLIO_REPO_OWNER` (env var, e.g., `cdcoonce`)
- `WAGA_PORTFOLIO_REPO_NAME` (env var, e.g., `charleslikesdata`)
- `WAGA_PORTFOLIO_REPO_TOKEN` (Dagster Cloud secret, fine-grained PAT)
- `WAGA_PORTFOLIO_REPO_BRANCH` (env var, default `main`)

**Asset check:** `waga_dashboard_export_commit_landed` verifies the commit SHA
resolves on the portfolio repo. Runs after the asset.

**Failure mode:** If the push fails, the asset fails. The local files are still
written, so a human can retry manually or reviewer can debug from the local
copy.

### 2. Panel App

**Location:** `src/weather_analytics/dashboard/`

**Structure:**

```
src/weather_analytics/dashboard/
  ├── __init__.py
  ├── app.py              # Entry point — builds the Panel layout
  ├── data_loader.py      # Fetches and caches JSON, returns Polars frames
  ├── components/
  │   ├── __init__.py
  │   ├── filters.py      # Asset selector, date range, type toggle
  │   ├── kpi_cards.py    # Top-row summary cards
  │   ├── fleet_view.py   # Fleet overview tab contents
  │   ├── asset_view.py   # Asset deep-dive tab contents
  │   └── weather_view.py # Weather correlation tab contents
  ├── theme.py            # Bokeh Theme + CSS variables
  └── static/
      └── portfolio.css   # Poppins import, palette, card styling
```

**Module responsibilities:**

| Module                       | Responsibility                                                        | Interface                                                                                    |
| ---------------------------- | --------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| `app.py`                     | Assemble the `FastListTemplate`, wire components, expose servable app | `servable()` entrypoint for `panel convert`                                                  |
| `data_loader.py`             | Fetch JSON files, return Polars DataFrames, cache results             | `load_manifest()`, `load_assets()`, `load_daily_performance()`, `load_weather_performance()` |
| `components/filters.py`      | Build filter widgets, expose as Panel params                          | `Filters` class with `asset_id`, `date_range`, `asset_type` params                           |
| `components/kpi_cards.py`    | Summary cards at the top of the page                                  | `kpi_row(filtered_df) -> pn.Row`                                                             |
| `components/fleet_view.py`   | Fleet overview charts (generation, CF, performance heatmap)           | `fleet_panel(filters) -> pn.Column`                                                          |
| `components/asset_view.py`   | Single-asset drill-down charts                                        | `asset_panel(filters) -> pn.Column`                                                          |
| `components/weather_view.py` | Wind/solar correlation plots                                          | `weather_panel(filters) -> pn.Column`                                                        |
| `theme.py`                   | Global Bokeh Theme object + CSS string                                | `portfolio_theme` object, `portfolio_css` string                                             |

Each `_view.py` file uses `@pn.depends()` to react to filter changes without
knowing about other components. Components are independently testable.

### 3. Layout

Visitor experience, top to bottom:

1. **Header** — project title, one-sentence description, link to WAGA GitHub
   repo, last-updated timestamp from manifest
2. **Filter bar** — asset selector (single-select with "All" option), date
   range picker, asset type toggle (All / Wind / Solar)
3. **KPI row** — four cards: Total MWh, Avg Capacity Factor, Avg Availability,
   Avg Performance Score
4. **Tabs:**
   - **Fleet Overview** — generation over time (stacked by asset), capacity
     factor by asset (bar chart), performance score heatmap (asset × date)
   - **Asset Deep-Dive** — daily generation with expected vs. actual overlay,
     rolling 7d/30d capacity factor, wind-or-solar scatter with regression
     line, performance distribution hours (stacked bar)
   - **Weather Correlation** — wind/solar r² matrix by asset, wind speed vs.
     generation scatter, GHI vs. generation scatter
5. **Footer** — data freshness, "built with Panel + Dagster + Snowflake + dbt",
   link to GitHub repo

## Data Contract

### Export format decisions

- **JSON**, not Parquet — Pyodide fetches natively, no extra library needed
- **Denormalized** — one row per asset-day, no joins needed in browser
- **Projected** — only columns the UI uses (~15 per mart), not full mart schema
- **ISO date strings** — easier to debug than epoch seconds
- **Schema version field** in manifest, enforced as warning banner on mismatch

### File: `manifest.json`

```json
{
  "generated_at": "2026-04-11T18:30:00Z",
  "pipeline_run_id": "abc123",
  "date_range": {
    "start": "2025-01-01",
    "end": "2026-04-10"
  },
  "asset_count": 10,
  "row_counts": {
    "daily_performance": 3650,
    "weather_performance": 3650
  },
  "schema_version": "1.0"
}
```

### File: `assets.json`

One record per asset:

```json
[
  {
    "asset_id": "ASSET_001",
    "asset_type": "wind",
    "capacity_mw": 50,
    "size_category": "Small",
    "display_name": "Wind Asset 001 (50 MW)"
  }
]
```

### File: `daily_performance.json`

Projected from `mart_asset_performance_daily`. One record per asset-day:

```json
[
  {
    "asset_id": "ASSET_001",
    "date": "2026-04-10",
    "total_net_generation_mwh": 487.3,
    "daily_capacity_factor": 0.406,
    "avg_availability_pct": 96.2,
    "total_curtailment_mwh": 12.1,
    "daily_performance_rating": "High",
    "excellent_hours": 18,
    "good_hours": 4,
    "fair_hours": 2,
    "poor_hours": 0,
    "avg_wind_speed_mps": 8.3,
    "avg_ghi": 245.0,
    "avg_temperature_c": 14.5,
    "data_completeness_pct": 100.0
  }
]
```

### File: `weather_performance.json`

Projected from `mart_asset_weather_performance`. One record per asset-day:

```json
[
  {
    "asset_id": "ASSET_001",
    "date": "2026-04-10",
    "performance_score": 87.3,
    "performance_category": "Excellent",
    "avg_expected_generation_mwh": 512.0,
    "avg_actual_generation_mwh": 487.3,
    "avg_performance_ratio_pct": 95.2,
    "wind_r_squared": 0.82,
    "solar_r_squared": 0.04,
    "inferred_asset_type": "wind",
    "rolling_7d_avg_cf": 0.389,
    "rolling_30d_avg_cf": 0.371
  }
]
```

### Size targets

| File                       | Raw         | Gzipped    |
| -------------------------- | ----------- | ---------- |
| `manifest.json`            | ~500 bytes  | ~300 bytes |
| `assets.json`              | ~1 KB       | ~600 bytes |
| `daily_performance.json`   | ~250 KB     | ~50 KB     |
| `weather_performance.json` | ~220 KB     | ~45 KB     |
| **Total**                  | **~470 KB** | **~95 KB** |

GitHub Pages serves gzip by default, so visitors download ~100 KB. Pyodide
runtime (~10 MB) is browser-cached after first visit.

### Scaling headroom

At current scale (~3,650 rows/year × 15 columns), JSON is both simpler and
fast enough. If the fleet grows to ~1,000 assets × 5 years, the export format
should swap to Parquet + `pyarrow` in Pyodide. The export asset should be
written so this swap is a single-file change.

## Theming

Four-layer theme strategy to match charleslikesdata.com.

### Layer 1: Portfolio CSS

`static/portfolio.css` — imports Poppins, declares CSS custom properties
matching the portfolio's exact palette, styles Panel cards to match the
portfolio's project-card style (border-radius: 10px, box-shadow matching
portfolio), styles filter bar and KPI cards.

Palette copied verbatim from portfolio's `style.css`:

```text
:root {
  --color-text-primary: #353535;
  --color-text-secondary: rgb(85, 85, 85);
  --color-bg-white: #fff;
  --color-bg-light: #f9f9f9;
  --color-bg-card-highlight: rgba(235, 241, 248, 0.4);
  --color-border: rgb(53, 53, 53);
  --color-shadow: rgba(0, 0, 0, 0.1);
}
```

Font: `Poppins` weights 300/400/500/600 via Google Fonts import.

### Layer 2: Bokeh Theme

`theme.py` exports a `bokeh.themes.Theme` object that applies globally via
`pn.config.theme`. Sets:

- Figure background: white
- Outline: none
- Axis labels and ticks: Poppins, `#353535`
- Gridlines: `#f0f0f0` dashed
- Title: Poppins, weight normal, `#353535`
- Legend: transparent background, no border, Poppins
- Toolbar: logo removed, all tools hidden by default (re-enable later if
  wanted)

### Layer 3: Data palette

Desaturated, editorial. Used for encoding data, not decoration.

| Usage                         | Hex       |
| ----------------------------- | --------- |
| Primary data series           | `#353535` |
| Secondary / comparison        | `#a8b8c8` |
| Wind assets                   | `#4a7c7e` |
| Solar assets                  | `#d4a44c` |
| Performance: Excellent        | `#4a7c5c` |
| Performance: Good             | `#7ca087` |
| Performance: Fair             | `#c8a472` |
| Performance: Poor             | `#b06a5c` |
| Rolling / trend lines         | `#888`    |
| Expected generation (ghosted) | `#ccc`    |

### Layer 4: Widget styling

Targeted CSS overrides for Panel widgets:

- Dropdowns and inputs: Poppins, pill-shaped (2rem border-radius), charcoal
  border
- Tabs: Poppins weight 500, active tab underlined with 2px charcoal border
- Date picker: use browser-native date input in Pyodide mode

### Known imperfections

- Bokeh toolbar chrome is hard to fully theme. Hiding it by default sidesteps
  the issue.
- Tooltip styling inherits some Bokeh defaults. Good enough, not pixel-perfect.

## Deployment

### Flow A: Data refresh (every pipeline run)

Triggered by the existing Dagster daily schedule. No new schedule needed.

```
Dagster Cloud daily schedule
  └─> waga_dashboard_export asset runs
        └─> Query marts, project columns, build JSON payloads
              └─> Write to local dashboard_exports/ (debugging copy)
                    └─> Push to portfolio repo via GitHub Contents API
                          └─> Target: dashboard/data/*.json
                                └─> Single commit with all four files
                                      └─> Commit message: "chore(dashboard): refresh data YYYY-MM-DD [pipeline run abc123]"
                                            └─> GitHub Pages auto-rebuilds (1-2 min)
                                                  └─> Live dashboard reflects new data
```

**Auth:** Fine-grained PAT stored as `WAGA_PORTFOLIO_REPO_TOKEN` in Dagster
Cloud secrets. Scoped to `contents: write` on the portfolio repo only. Never
committed anywhere.

**Idempotency:** The Contents API handles identical content as a no-op or
replaces on SHA mismatch — running the pipeline twice in one day produces
either one commit or two identical commits, never corruption.

### Flow B: App code refresh (automated via GitHub Action)

Triggered by pushes to `main` in the WAGA repo that touch
`src/weather_analytics/dashboard/**`.

```
Push to main (with changes under src/weather_analytics/dashboard/)
  └─> .github/workflows/build-dashboard.yml fires
        └─> Checkout WAGA repo
              └─> uv sync (install Panel, Bokeh, etc.)
                    └─> uv run panel convert src/weather_analytics/dashboard/app.py \
                          --to pyodide-worker --out dashboard_build/
                          └─> Copy static/portfolio.css into dashboard_build/
                                └─> Use WAGA_PORTFOLIO_REPO_TOKEN to commit dashboard_build/*
                                      └─> Target: <portfolio-repo>/dashboard/
                                            └─> Commit message: "chore(dashboard): rebuild app [WAGA commit SHA]"
                                                  └─> GitHub Pages rebuild
```

**Workflow file:** `.github/workflows/build-dashboard.yml`

**Path filter:** only runs on changes under `src/weather_analytics/dashboard/**`
or `static/portfolio.css`. Avoids rebuilding on unrelated commits.

**Same PAT** as Flow A — `WAGA_PORTFOLIO_REPO_TOKEN` stored as a GitHub Actions
secret in the WAGA repo.

### Flow C: First-time setup (one-time)

1. Create GitHub fine-grained PAT scoped to portfolio repo, `contents: write`.
2. Store PAT as `WAGA_PORTFOLIO_REPO_TOKEN` in:
   - Dagster Cloud secrets (for Flow A)
   - WAGA repo GitHub Actions secrets (for Flow B)
   - Local `.env` (for local testing)
3. Add `panel`, `bokeh`, `pygithub` to `pyproject.toml` dependencies.
4. Create empty `dashboard/` directory in portfolio repo (GitHub Pages serves it).
5. Add dashboard link to portfolio site navigation (`<a href="/dashboard/">Dashboard</a>`).
6. Implement Dagster asset + Panel app code (covered in implementation plan).
7. Run first build and push manually to verify end-to-end before relying on
   automation.

## Testing Strategy

Per the existing WAGA test conventions (pytest markers `unit`, `integration`):

- **Unit tests** for:
  - `data_loader.py` — given a JSON payload, returns correctly typed Polars frame
  - `components/filters.py` — filter state mutations
  - `components/kpi_cards.py` — given a DataFrame, computes expected KPI values
  - `dashboard_export.py` — column projection correctness, JSON serialization
    shape, manifest metadata

- **Integration tests** for:
  - `dashboard_export.py` against the real marts (mocked Snowflake connection,
    real dbt fixture data if available)
  - GitHub Contents API push (mocked PyGithub)

- **Manual verification** for:
  - `panel convert` build produces valid output
  - Pyodide loads and runs in-browser
  - Theme renders correctly in Chrome and Safari
  - Dashboard looks right on desktop (primary target) and acceptable on mobile

- **No tests for:**
  - Panel/Bokeh internal rendering (trust upstream)
  - GitHub Actions workflow itself (test by running it)

## Cost

All components are open source or already paid for:

- Panel, Bokeh, Pyodide: free (BSD/MIT)
- GitHub Pages: free (already using)
- GitHub PAT: free
- Dagster Cloud: already paying, no marginal cost
- Network egress: negligible
- Storage: <1 MB in portfolio repo

**Total new cost: $0/month.**

## Risks & Mitigations

| Risk                                             | Mitigation                                                                                                                                                        |
| ------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| PAT leaks                                        | Fine-grained scope, stored only in secrets, rotate quarterly                                                                                                      |
| Dagster push fails silently                      | Asset raises `dagster.Failure` on any API error, asset check verifies commit SHA landed                                                                           |
| Panel/Bokeh version upgrade breaks Pyodide build | Pin `panel` and `bokeh` versions in `pyproject.toml`, run build in CI before promoting                                                                            |
| Data file exceeds browser comfortable size       | Manifest's `row_counts` checked at app startup, warning banner if over threshold (e.g., 50k rows)                                                                 |
| Portfolio repo gets spammed with auto-commits    | One commit per pipeline run (all data files bundled), one commit per app code push                                                                                |
| Pyodide incompatible with a library added later  | Restrict Panel app dependencies to known-Pyodide-compatible set: `panel`, `bokeh`, `polars`, `numpy`. Document this constraint in the dashboard module docstring. |
| Schema version mismatch between export and app   | Warning banner (non-blocking), not a crash                                                                                                                        |
